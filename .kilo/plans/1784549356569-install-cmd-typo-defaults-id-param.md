# Fixes: install command typo, defaults, and per-id installer support

## Context
PR #4 introduced an install modal + generated commands. The user reported concrete defects found in the generated PowerShell command and requested behavior/UX changes:

1. **Typo in generated PowerShell command** â€” `build-install-cmd.ts` emits `install-${meta.scriptPs1}`, producing `install-install-agent.ps1` (double `install-`). Should be `${meta.scriptPs1}` (`install-agent.ps1`).
2. **Default target path** â€” "path to project" should default to `.` (current dir), not `/path/to/project` / `C:\path\to\project`.
3. **Default tool selection** â€” all tool checkboxes are currently `checked` by default; they must be **unselected** by default (modal already disables Copy when 0 tools selected).
4. **Per-item install** â€” install scripts must accept a skill/agent **identifier** param (`--id` / `-Id`). In the FE, when the modal is opened from a skill/agent **detail page**, the id field is pre-filled and **readonly**.

Skill/agent folders are keyed by slug equal to the folder name (e.g. `skills/backoffice-design`, `agents/bookworm`), so `--id backoffice-design` maps directly to that subfolder.

## Affected files
- `_astro/src/lib/build-install-cmd.ts` (typo + default target + id param)
- `_astro/src/components/InstallModal.astro` (uncheck tools + readonly id input + id wiring)
- `_astro/src/components/InstallCTA.astro` (pass id to modal)
- `_astro/src/pages/skills/[name].astro`, `_astro/src/pages/agents/[name].astro` (pass `id={item.slug}`)
- `install-skill.sh`, `install-agent.sh` (root, canonical)
- `install-skill.ps1`, `install-agent.ps1` (root, canonical)
- `_astro/public/install-*.{sh,ps1}` (regenerated by `scripts/sync-installers.mjs` on `prebuild`)
- `_astro/scripts/sync-installers.mjs` (assert tool lists â€” unchanged, but re-runs on build)

All installer edits happen in the **root** files; public copies are overwritten at build time by the sync script. Do not hand-edit `_astro/public/`.

## Changes

### 1. `build-install-cmd.ts`
- **Fix typo**: line ~101 `install-${meta.scriptPs1}` â†’ `${meta.scriptPs1}`.
- **Add `id?` to `BuildOptions`** and thread it through `buildCommand`.
- **Default target** when `scope === 'project'` and `target` is empty/whitespace â†’ `'.'` (both platforms). Remove the `/path/to/project` / `C:\path\to\project` placeholders.
- **Emit `-Id`/`--id`**:
  - Windows: `pwsh .\${meta.scriptPs1} -Tool ${tools}` â†’ append `-Id ${quoteWindows(id)}` when `id` present.
  - Unix: `bash /tmp/${meta.scriptBash} --tool ${tools}` â†’ append `--id ${quoteUnix(id)}` when `id` present.
  - Order: tool, then target/global, then id (id last is fine).
- Keep quoting helpers (`quoteUnix`, `quoteWindows`) â€” reuse for id.

### 2. `InstallModal.astro`
- Tools checkboxes: remove `checked` from `<input type="checkbox" ...>` (line ~55) so **none selected by default**.
- Add an **Identifier** field in the modal markup:
  - `<input type="text" data-install-id placeholder="skill or agent id (optional)" />`
  - Accept a `Props.id?: string`. When provided: set `value={id}` and add `readonly` + a subtle disabled/readonly style; also do NOT show placeholder.
- In the `<script>`'s `render()`, read the id input value and pass `id: idInput.value` to `buildCommand`. Guard: empty/whitespace â†’ omit.
- Expose `idInput` via `root.querySelector('[data-install-id]')`.

### 3. `InstallCTA.astro`
- Add `Props.id?: string`, pass it through to `data-install-open` button via a `data-install-id` attribute OR (cleaner) keep the button as-is and have the page render the modal with the id. Simplest: add optional `id` prop and emit `data-install-id={id}` on the button; `InstallModal.openModal` reads it. 
  - **Decision**: `InstallCTA` already just triggers open. To pass id into the modal, set `data-install-id={id}` on the button, and in `openModal(kind)` read `btn.dataset.installId` and set it on the modal's `[data-install-id]` input (and toggle readonly) before showing. This avoids changing the modal's own `Props` and keeps the index page (no id) working unchanged.

### 4. Detail pages (`skills/[name].astro`, `agents/[name].astro`)
- `<InstallCTA kind="skill" id={item.slug} />` (and `agent`). The slug equals the folder name, so the id matches the installer's subfolder.

### 5. Installers (root `install-skill.sh` / `install-agent.sh`)
- Add `--id <name>` to arg parsing (and to `usage`).
- When `--id` is set: install **only** that single folder (`skills/<id>` / `agents/<id>`); if it does not exist, error `ERROR: no skill/agent with id '<id>'`. When omitted: current behavior (all folders).
- Implementation: derive the list of source dirs from `[[ -n "$ID" ]] && echo "$SKILLS_DIR/$ID" || ls -d "$SKILLS_DIR"/*/` â€” but keep the existing per-tool loop. The `--remove` path likewise filters to that id.
- Same change mirrored in `install-agent.sh` (`agents/<id>`).

### 6. Installers (root `install-skill.ps1` / `install-agent.ps1`)
- Add `[string] $Id` parameter + `.PARAMETER Id` doc + example.
- Filter `Get-ChildItem -LiteralPath $SkillsDir -Directory` to a single dir when `$Id` set; error if missing.
- Same in `install-agent.ps1` (`$AgentsDir`).

### 7. Sync script
- No change required. It only asserts tool-list drift; id param is orthogonal.

## Validation
- `npm --prefix _astro run build` (runs `prebuild` sync, regenerating `public/install-*.{sh,ps1}` and asserting tool lists) must pass.
- `npm --prefix _astro run check` â€” no new errors.
- Manual/command checks:
  - `bash _astro/public/install-skill.sh --tool claude --target . --id backoffice-design` â†’ links only `skills/backoffice-design`.
  - `bash _astro/public/install-skill.sh --tool claude --target .` (no id) â†’ links all skills.
  - `bash _astro/public/install-skill.sh --tool claude --id nope` â†’ errors, no install.
  - Windows command generated by modal for agent id `bookworm` reads `pwsh .\install-agent.ps1 -Tool kilo -Target "." -Id bookworm` (no `install-` duplicate).
  - Modal: no tool checked by default; target input default value `.`; opening from detail page shows id pre-filled + readonly; generated command includes `--id <slug>`.

## Risks / open questions
- **Readonly UX**: readonly input still focusable; acceptable. Could add `disabled` but then value won't submit â€” we keep `readonly` (value preserved).
- **Powershell symlink on Windows**: unchanged from current behavior.
- The index page (`InstallCTA` without id) leaves the field empty â†’ optional, all-items install.
